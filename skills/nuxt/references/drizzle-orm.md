# Drizzle ORM

Use this reference when adding or reviewing Drizzle ORM in a Nuxt project.

---

## Project Structure Rules

- Keep schema definitions in `server/database/schema/` — one file per domain.
- Keep the Drizzle client factory in `server/utils/drizzle.ts`.
- Keep generated migration files in `server/database/migrations/`.
- Keep `drizzle.config.ts` at the project root.
- Never import any database code outside `server/` — Nuxt will not bundle it on the client.

```
server/
├── api/                          ← Thin Nitro route handlers
├── database/
│   ├── schema/                   ← Table + relation definitions (one file per domain)
│   │   ├── users.ts
│   │   ├── posts.ts
│   │   └── index.ts              ← Re-exports all schema for drizzle config
│   └── migrations/               ← Auto-generated by drizzle-kit
├── plugins/
│   └── drizzle.server.ts         ← Nitro plugin: boot-time migration guard
└── utils/
    └── drizzle.ts                ← useDrizzle() singleton + tables export
drizzle.config.ts
.env
```

## Runtime Client Pattern — Singleton

Always create the Drizzle client once and reuse it. Never open a new connection per request.

**PostgreSQL (`server/utils/drizzle.ts`):**
```ts
import { drizzle } from 'drizzle-orm/postgres-js'
import postgres from 'postgres'
import * as schema from '../database/schema'

export const tables = schema

let _db: ReturnType<typeof drizzle> | null = null

export function useDrizzle() {
  if (!_db) {
    const { databaseUrl } = useRuntimeConfig()
    // max controls connection pool size — tune to your DB plan
    const client = postgres(databaseUrl, { max: 10 })
    _db = drizzle(client, { schema })
  }
  return _db
}
```

**MySQL (`server/utils/drizzle.ts`):**
```ts
import { drizzle } from 'drizzle-orm/mysql2'
import mysql from 'mysql2/promise'
import * as schema from '../database/schema'

export const tables = schema

let _db: ReturnType<typeof drizzle> | null = null

export function useDrizzle() {
  if (!_db) {
    const { databaseUrl } = useRuntimeConfig()
    const pool = mysql.createPool({
      uri: databaseUrl,
      connectionLimit: 10,
      connectTimeout: 5000,
    })
    _db = drizzle(pool, { schema, mode: 'default' })
  }
  return _db
}
```

**SQLite (`server/utils/drizzle.ts`):**
```ts
import { drizzle } from 'drizzle-orm/better-sqlite3'
import Database from 'better-sqlite3'
import * as schema from '../database/schema'

export const tables = schema

let _db: ReturnType<typeof drizzle> | null = null

export function useDrizzle() {
  if (!_db) {
    const sqlite = new Database('./dev.db')
    _db = drizzle(sqlite, { schema })
  }
  return _db
}
```

The singleton lives on the module level, not `globalThis`, so Nitro's module cache keeps it hot-reload safe in development and connection-stable in production.

## Nitro Plugin — Boot-Time Migration Guard

Run migrations automatically at server startup using a Nitro plugin. This prevents the migration from running on every hot-reload.

```ts
// server/plugins/drizzle.server.ts
import { migrate } from 'drizzle-orm/postgres-js/migrator'

export default defineNitroPlugin(async () => {
  try {
    const db = useDrizzle()
    await migrate(db, { migrationsFolder: 'server/database/migrations' })
    console.log('✅ Database migrations applied')
  }
  catch (err) {
    console.error('❌ Database migration failed:', err)
    throw err // fail fast — do not start the server with a broken schema
  }
})
```

> Skip this plugin in development when using `drizzle-kit push`. Enable it in staging and production where migration files are used.

## Schema Design Rules

### Use Identity Columns (PostgreSQL — 2025 standard)

PostgreSQL now recommends `integer().generatedAlwaysAsIdentity()` over `serial`. Always use this for new tables.

```ts
// server/database/schema/users.ts
import { pgTable, pgEnum, integer, varchar, text, timestamp, boolean, jsonb, index, uniqueIndex } from 'drizzle-orm/pg-core'
import { relations, sql } from 'drizzle-orm'

// ── Enums ──────────────────────────────────────────────────────────────────
export const userRoleEnum = pgEnum('user_role', ['admin', 'user', 'moderator'])

// ── Reusable timestamp columns ─────────────────────────────────────────────
export const timestamps = {
  createdAt: timestamp('created_at', { mode: 'date', precision: 3, withTimezone: true })
    .defaultNow().notNull(),
  updatedAt: timestamp('updated_at', { mode: 'date', precision: 3, withTimezone: true })
    .defaultNow().notNull().$onUpdateFn(() => new Date()),
}

// ── Table ──────────────────────────────────────────────────────────────────
export const users = pgTable('users', {
  // Identity column — the new PostgreSQL standard (not serial)
  id: integer('id').primaryKey().generatedAlwaysAsIdentity(),

  // Use varchar with length limits, not unbounded text, for indexed columns
  email: varchar('email', { length: 320 }).notNull().unique(),
  name: varchar('name', { length: 100 }).notNull(),
  role: userRoleEnum('role').default('user').notNull(),
  isActive: boolean('is_active').default(true).notNull(),

  // Type JSONB columns for TypeScript safety
  preferences: jsonb('preferences').$type<{
    theme: 'light' | 'dark'
    notifications: boolean
  }>(),

  ...timestamps,
}, (table) => [
  uniqueIndex('users_email_unique').on(table.email),
  // Partial index — only index active users, reducing index size
  index('users_active_email_idx').on(table.email).where(sql`${table.isActive} = true`),
])

// ── TypeScript types ───────────────────────────────────────────────────────
export type User = typeof users.$inferSelect
export type NewUser = typeof users.$inferInsert
```

### Timestamp Best Practices

- Always use `{ mode: 'date', precision: 3, withTimezone: true }` for timestamps.
- `mode: 'date'` is 10–15% faster than `mode: 'string'`. Use string mode only when you need raw formatting control.
- Use `$onUpdateFn(() => new Date())` on `updatedAt` for automatic updates.

### Indexing Rules

- Always index foreign key columns to avoid sequential scans on joins.
- Use composite indexes when queries filter on multiple columns — column order matters (put the most selective column first).
- Use partial indexes for queries that always include a fixed `WHERE` clause (e.g., `WHERE status = 'active'`).
- Use covering indexes (`.include(col)`) to avoid table lookups for hot queries.
- Use GIN indexes for `jsonb` and `text[]` columns.
- Avoid over-indexing: each index slows writes and increases storage. Add indexes when query analysis confirms a bottleneck.

```ts
// server/database/schema/posts.ts
import { pgTable, pgEnum, integer, varchar, text, timestamp, index, sql } from 'drizzle-orm/pg-core'
import { relations } from 'drizzle-orm'
import { users, timestamps } from './users'

export const postStatusEnum = pgEnum('post_status', ['draft', 'published', 'archived'])

export const posts = pgTable('posts', {
  id: integer('id').primaryKey().generatedAlwaysAsIdentity(),
  authorId: integer('author_id')
    .references(() => users.id, { onDelete: 'cascade', onUpdate: 'cascade' })
    .notNull(),
  title: varchar('title', { length: 255 }).notNull(),
  content: text('content'),
  status: postStatusEnum('status').default('draft').notNull(),
  publishedAt: timestamp('published_at', { mode: 'date', precision: 3, withTimezone: true }),
  ...timestamps,
}, (table) => [
  // Always index foreign keys
  index('posts_author_idx').on(table.authorId),
  // Composite index: user's published posts, ordered by date
  index('posts_author_status_idx').on(table.authorId, table.status, table.publishedAt.desc()),
  // Partial index: only published posts appear in public listings
  index('posts_published_idx').on(table.publishedAt.desc()).where(sql`${table.status} = 'published'`),
])

export type Post = typeof posts.$inferSelect
export type NewPost = typeof posts.$inferInsert
```

### Relations

Define Drizzle relations alongside your table definitions. Relations power the query API (`db.query.*`).

```ts
// Relations for the query API
export const usersRelations = relations(users, ({ many }) => ({
  posts: many(posts),
}))

export const postsRelations = relations(posts, ({ one }) => ({
  author: one(users, {
    fields: [posts.authorId],
    references: [users.id],
  }),
}))
```

### Schema Index Re-export

```ts
// server/database/schema/index.ts
export * from './users'
export * from './posts'
```

## Schema Change Workflow

| Stage | Command | Notes |
|---|---|---|
| Development | `npx drizzle-kit push` | Pushes schema directly, no migration files |
| Staging / Production | `npx drizzle-kit generate` then `npx drizzle-kit migrate` | Creates versioned migration files |
| Inspect DB | `npx drizzle-kit studio` | Visual browser for the database |
| Sync types | `npx drizzle-kit pull` | Generates schema from an existing database |

Never run `push` against a production database. Use migration files with a CI/CD step or the Nitro boot plugin.

## Querying — Relational API vs Select

### Prefer the Relational Query API for related data

Use `db.query.<table>.findMany()` when you need to load relations. It generates precise TypeScript types, supports selective column loading, and avoids manual joins.

```ts
// Load posts with their author — selective columns, no over-fetching
const posts = await db.query.posts.findMany({
  columns: { id: true, title: true, publishedAt: true },
  with: {
    author: { columns: { id: true, name: true } },
  },
  where: (posts, { eq }) => eq(posts.status, 'published'),
  orderBy: (posts, { desc }) => desc(posts.publishedAt),
  limit: 20,
  offset: 0,
})
```

### Use raw select for simple or performance-critical queries

```ts
import { eq, and, desc } from 'drizzle-orm'

const user = await db
  .select()
  .from(tables.users)
  .where(eq(tables.users.id, id))
  .then(r => r[0] ?? null)
```

### Use prepared statements for hot paths

```ts
// Define once — reuse on every request
const getUserByEmail = db
  .select()
  .from(tables.users)
  .where(eq(tables.users.email, sql.placeholder('email')))
  .prepare('get_user_by_email')

// Execute with parameters
const user = await getUserByEmail.execute({ email: 'alice@example.com' })
```

## Input Validation with Zod (drizzle-zod)

Always validate incoming request bodies before passing data to Drizzle. Use `drizzle-zod` to generate Zod schemas directly from table definitions so validation and schema stay in sync.

```ts
// server/database/schema/users.ts — add alongside the table
import { createInsertSchema, createSelectSchema } from 'drizzle-zod'
import { z } from 'zod'

export const insertUserSchema = createInsertSchema(users, {
  email: schema => schema.email('Invalid email format'),
  name: schema => schema.min(2, 'Name must be at least 2 characters'),
})

export const selectUserSchema = createSelectSchema(users)

// A stricter schema for the API surface (omit server-set fields)
export const createUserInputSchema = insertUserSchema.omit({ id: true, createdAt: true, updatedAt: true })
export type CreateUserInput = z.infer<typeof createUserInputSchema>
```

Use the schema in API routes:

```ts
// server/api/users/index.post.ts
export default defineEventHandler(async (event) => {
  const raw = await readBody(event)
  const body = createUserInputSchema.parse(raw) // throws H3Error on invalid input

  const db = useDrizzle()
  const [user] = await db.insert(tables.users).values(body).returning()
  return user
})
```

For cleaner error handling, wrap `parse` with a try/catch and map Zod errors to a 422 response:

```ts
import { z } from 'zod'

export default defineEventHandler(async (event) => {
  const raw = await readBody(event)

  let body: CreateUserInput
  try {
    body = createUserInputSchema.parse(raw)
  }
  catch (err) {
    if (err instanceof z.ZodError) {
      throw createError({ statusCode: 422, data: err.flatten() })
    }
    throw err
  }

  const db = useDrizzle()
  const [user] = await db.insert(tables.users).values(body).returning()
  return user
})
```

## Data-Access Layer Pattern

As the project grows, pull Drizzle queries out of route handlers and into a repository module. Route handlers should only validate input, authorize, call the repository, and return the response.

```ts
// server/utils/repositories/userRepository.ts
import { eq } from 'drizzle-orm'
import type { CreateUserInput } from '../database/schema/users'

export async function findUsers() {
  return useDrizzle().select().from(tables.users)
}

export async function findUserById(id: number) {
  const db = useDrizzle()
  const [user] = await db.select().from(tables.users).where(eq(tables.users.id, id))
  return user ?? null
}

export async function createUser(data: CreateUserInput) {
  const db = useDrizzle()
  const [user] = await db.insert(tables.users).values(data).returning()
  return user
}

export async function deleteUser(id: number) {
  const db = useDrizzle()
  await db.delete(tables.users).where(eq(tables.users.id, id))
}
```

## Using Drizzle in API Routes

Keep handlers thin — validate, authorize, delegate, respond.

```ts
// server/api/users/index.get.ts
export default defineEventHandler(async () => {
  return findUsers()
})

// server/api/users/index.post.ts
export default defineEventHandler(async (event) => {
  const body = createUserInputSchema.parse(await readBody(event))
  return createUser(body)
})

// server/api/users/[id].get.ts
export default defineEventHandler(async (event) => {
  const id = Number(getRouterParam(event, 'id'))
  const user = await findUserById(id)
  if (!user) throw createError({ statusCode: 404, message: 'User not found' })
  return user
})

// server/api/users/[id].delete.ts
export default defineEventHandler(async (event) => {
  const id = Number(getRouterParam(event, 'id'))
  await deleteUser(id)
  return { success: true }
})
```

## Transactions

Use transactions when multiple writes must succeed or fail together.

```ts
import { eq } from 'drizzle-orm'

export async function transferPost(postId: number, newAuthorId: number) {
  const db = useDrizzle()
  return db.transaction(async (tx) => {
    const [post] = await tx
      .select()
      .from(tables.posts)
      .where(eq(tables.posts.id, postId))
    if (!post) throw new Error('Post not found')

    await tx
      .update(tables.posts)
      .set({ authorId: newAuthorId })
      .where(eq(tables.posts.id, postId))

    return post
  })
}
```

## Seeding with Nitro Tasks

```ts
// server/tasks/db/seed.ts
export default defineTask({
  meta: { name: 'db:seed', description: 'Seed the database with initial data' },
  async run() {
    const db = useDrizzle()
    await db.insert(tables.users).values([
      { email: 'alice@example.com', name: 'Alice Smith', role: 'admin' },
      { email: 'bob@example.com', name: 'Bob Jones', role: 'user' },
    ]).onConflictDoNothing()
    return { result: 'Seeded successfully' }
  },
})
```

Enable Nitro tasks in `nuxt.config.ts`:

```ts
export default defineNuxtConfig({
  nitro: {
    experimental: { tasks: true },
  },
})
```

Run: `npx nitro task run db:seed`

## Mapping Pattern

Do not leak raw ORM shapes to the frontend. Map records when you need to:

- Normalize timestamps to ISO strings.
- Hide internal fields (e.g., password hashes, internal IDs).
- Produce a stable, versioned API shape that can evolve independently of the schema.
- Combine fields from multiple tables into a single response object.

## Connection Security

| Practice | How |
|---|---|
| Keep credentials server-only | Use `runtimeConfig` private keys, never public |
| Least privilege | Only grant SELECT/INSERT/UPDATE/DELETE — never root |
| TLS/SSL | Enable SSL in the driver options for remote DBs |
| Connection pooling | Set `max` in `postgres-js` or `connectionLimit` in mysql2 |
| No raw SQL in logs | Never log full query strings in production |

## Common Pitfalls to Avoid

- **Opening a new DB client per request** — always use the singleton pattern.
- **Using `serial` for primary keys** — use `generatedAlwaysAsIdentity()` instead (PostgreSQL).
- **Using `mode: 'string'` for timestamps** — prefer `mode: 'date'` for performance.
- **Missing indexes on foreign keys** — every FK column should have an index.
- **Over-indexing** — each index has a write cost; only add indexes that query analysis justifies.
- **Importing DB code in pages or components** — keep all DB code inside `server/`.
- **Running `drizzle-kit push` in production** — use migration files in production.
- **Skipping Zod validation** — always parse and validate request bodies before querying.
- **Leaking raw ORM types to the API** — map response shapes explicitly.

---

## Lessons to Carry Forward

- The singleton `useDrizzle()` + exported `tables` + a repository layer is the cleanest pattern at scale.
- `drizzle-zod` keeps validation in sync with the schema with zero duplication.
- The relational query API (`db.query.*`) produces better types and cleaner code than hand-written joins for most cases.
- Prepared statements are a cheap win for high-frequency queries.
- Use the Nitro boot plugin only for staging/production auto-migration; keep development fast with `push`.
- Identity columns, precise timestamps, and partial indexes are the 2025 PostgreSQL defaults.
